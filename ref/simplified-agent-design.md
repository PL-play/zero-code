# 极简极致 Agent 设计方案

## 设计哲学

> **简单 ≠ 简陋，极简 = 极致的工程打磨**

本方案基于以下核心信念：
- 单线程 + 主循环 + 工具管线的模式足够构建强大的 Code Agent
- 关键在于把核心路径打磨到**干净、稳定、可观测**
- 不需要工业级的复杂度，但需要**工业级的严谨**

**核心隐喻**：你现在的模式可以看成一个很小的"操作系统"——
- 一个事件循环
- 一组系统调用（tools/skills）
- 一点内存管理

要做到"极致"，关键不是加复杂度，而是把几条主干设计打磨到非常干净、稳定、可观测。

## 二、主循环：显式状态机

### 2.1 核心理念

**不要写成一坨 `while(true)`，而是显式状态机。**

一次「对话轮次」建模成有限状态机，每个状态是一个独立函数，主循环只是状态转移：
```
currentState = step(currentState, context)
```

### 2.2 关键状态设计

典型状态划分：
- **INIT** - 初始化上下文
- **COMPRESS_CHECK** - 检查是否需要压缩上下文
- **BUILD_SYSTEM_PROMPT** - 构建系统提示
- **THINKING** - LLM 思考中（流式输出）
- **TOOL_DECISION** - 解析并决定调用哪些工具
- **TOOL_EXEC** - 执行工具管线
- **POST_PROCESS** - 后处理与状态更新
- **DONE / ERROR** - 结束或错误状态

### 2.3 关键设计决策

**决策 1：行为可预测**
- 每个状态有明确的输入、输出、副作用
- 日志可以精确记录「当前卡在哪个状态」
- 便于调试：一眼看出问题发生在压缩阶段还是工具执行阶段

**决策 2：易于扩展**
- 插入新逻辑（如额外的安全检查）只需增加一个状态或修改现有状态
- 不需要改动整个循环结构

**决策 3：流式输出保持"状态感知"**
- 在 THINKING 状态按 chunk 输出，但每个 chunk 带上标记：
  - 普通内容
  - `tool_calls`（工具调用意图）
  - `planning`（思路梳理）
- 单线程也可以：先把"概要"和"计划"尽早流出，再慢慢补充实现细节

## 三、上下文压缩：结构化 + 可控

### 3.1 核心理念

**不求花哨的压缩算法，但求"结构化 + 可控"。**

### 3.2 三层记忆架构

```
┌─────────────────────────────────────────────────────────────┐
│                    短期记忆层                                 │
│  messages[] - 当前会话实时消息数组                            │
│  特征：O(1) 访问，完整保留当前对话细节                       │
└──────────────────────┬────────────────────────────────────┘
                       │ 92% 阈值触发
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                    中期记忆层                                 │
│  结构化压缩摘要（4 段式）                                      │
│  特征：智能压缩，保留关键信息，大幅节省 Token                 │
└──────────────────────┬────────────────────────────────────┘
                       │ 持久化触发
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                    长期记忆层                                 │
│  CLAUDE.md / 项目配置 / 用户偏好                              │
│  特征：跨会话恢复，项目持续记忆                               │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 关键实现方案

**方案 1：明确 Token 预算 + 三个阈值**

假设总上限 16k token：
- **60%**：开始提示"上下文偏多，建议你注意"
- **80%**：提醒 + 优先压缩"已结束任务"的历史对话
- **92%**：强制压缩（类似 Claude 的策略）

**方案 2：结构化摘要，而非纯 free-form**

借鉴 Claude Code 的 AU2 算法，简化为 **4 段式**：
1. **背景 / 目标** - 项目类型、核心目标
2. **关键决策 / 原因** - 重要技术选择及理由
3. **重要工具结果** - 生成的文件、关键执行结果
4. **未解决问题 / 下一步** - 待办事项、后续计划

使用固定模板 prompt，每次压缩都输出这个结构：
- 压缩结果对后续 LLM 提示更友好
- 模型更容易在 summary 上继续推理

**方案 3：按"任务块"压缩，而非按轮次截断**

- 给每个明显独立的任务分配 `task_id`
- 需要压缩时：
  - 优先把**已经结束的 task** 的历史轮次压成一个整体 summary
  - 保留当前活跃 task 的细节对话
- 这比简单按时间顺序砍掉旧消息聪明很多

## 四、工具系统：统一 6 阶段管线

### 4.1 核心理念

**不要"随便 call 函数"，而是走统一的 6 阶段管线。**

可以对齐 Claude Code 的 MH1 设计，但实现上保持简洁。

### 4.2 六阶段管线

```
┌─────────────────────────────────────────────────────────────┐
│  阶段 1：工具发现 & 绑定                                      │
│  从 tool_calls.name 映射到本地 Tool 对象                     │
│  包含：schema、execute、permissions 等元信息                 │
└──────────────────────┬────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 2：参数验证                                             │
│  所有工具统一用 schema（Zod / JSON Schema）做参数校验         │
│  错误返回统一格式                                             │
└──────────────────────┬────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 3：权限检查                                             │
│  支持三态：allow / deny / ask                                 │
│  ask → 在对话里插入确认问题，等待用户响应后继续               │
└──────────────────────┬────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 4：取消检查 & 超时控制                                  │
│  对外部调用（bash / http）设置超时                            │
│  全局 abort flag，支持用户中断                                │
└──────────────────────┬────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 5：执行                                                 │
│  所有工具走统一入口 executeTool(tool, input, context)       │
│  工具内部可 sync/async，对外暴露统一 interface                │
└──────────────────────┬────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 6：结果格式化 & 清理                                    │
│  输出统一格式：{ type, toolName, payload, error? }          │
│  用于：统一日志、统一注入 messages、统一统计                 │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 关键设计决策

**决策 1：配置级成本**
- 只要管线足够统一和稳定，以后加任何新工具几乎是"配置级成本"
- 只需要定义 schema + execute 函数 + 权限配置

**决策 2：错误处理标准化**
- 每个阶段的错误都有明确类型和恢复策略
- 可以统一记录、统一监控

## 五、SubAgent 管理：限制用途 + 严格生命周期

### 5.1 核心理念

**不要"乱套娃"，而是限制用途 + 严格生命周期。**

### 5.2 SubAgent 只做两类事

**类型 1：专项分析任务**
- 例如"只负责前端架构审查"的 Agent
- 有自己的 system prompt + 工具子集

**类型 2：长流程任务**
- 例如"跑完整测试 + 生成修复建议"的 Agent
- 有明确的开始和结束

**避免**：什么都丢给 SubAgent，否则上下文会过载。

### 5.3 严格生命周期管理

**创建时记录**：
- `parent_turn_id` - 父对话 ID
- `description` - 简短描述（3-5 词）
- `allowed_tools` - 可用工具白名单
- `max_steps` - 最大执行步数

**结束时强制输出结构化 summary**：
- `conclusion` - 核心结论
- `key_decisions` - 关键决策
- `tool_highlights` - 工具使用亮点
- `open_issues` - 遗留问题

**主 Agent 只接收这个 summary**，而不是把整段子对话并回主上下文：
- 极大减轻上下文压力
- 摘要本身更"抗压"（经过 LLM 提炼）

### 5.4 限制嵌套深度

- SubAgent 内禁止再启 SubAgent（或全局最多 N 个活跃）
- 即使单线程，这种逻辑上的约束能减少系统复杂度

## 六、工具与技能选择策略

### 6.1 核心理念

**不需要复杂 Planner，一个小而硬的 heuristic 就够用。**

### 6.2 三条核心规则

**规则 1：优先使用只读工具**

当模型"不太确定"时：
- 优先 `Read / Grep / LS` 收集证据
- 再决定是否 `Write / Edit / Bash`

在 system prompt 中明确写入：
> "如果你不确定，请优先使用只读工具去确认信息，然后再做写操作。"

**规则 2：显式鼓励"批量操作工具"**

类似 Claude 的 `MultiEdit` 使用风格：
- 当多个文件 / 多处改动高度相似时，鼓励生成一个批量编辑请求
- 即使实现只是顺序执行每个 edit，接口层面对 LLM 暴露的是"一个 multi-edit 调用"
- 更利于模型规划和理解

**规则 3：为典型长流程封装 Skill**

例如：
- "创建新页面 + 路由 + 测试 + 文档"
- "为函数写测试 → 跑测试 → 修失败 → 重跑"

Skill 内部仍然是多步工具 + 若干回合思考，但对主 Agent 来说就是一次调用：
- 更清晰的抽象
- 可以单独优化 skill 的 prompt / 策略
- 统计与调优粒度更合理

## 七、稳定性与可观测性

### 7.1 核心理念

**用日志和度量把"简单"变成"可调"。**

即使单线程，也可以做到很专业的可观测性。

### 7.2 关键日志维度

**每个 turn**：
- `turn_id`
- `user_intent`（粗粒度提取）
- `tools_called`
- `tokens_used`
- `compression_events`
- `errors`

**每次工具调用**：
- `tool_name`
- `input_size`
- `duration_ms`
- `success / error`
- 涉及 `file_paths`

**每次压缩**：
- `before_tokens`
- `after_tokens`
- `compression_ratio`
- `summary_length`

### 7.3 数据驱动的优化

有了这些日志，可以做：
1. **找出经常失败的工具 / prompt**，集中优化
2. **分析哪些场景频繁触发压缩**，反向调整上下文策略
3. **观察哪些 skill / pattern 效果最好**，反过来引导 system prompt 设计

## 八、扩展挂钩（Hooks）

### 8.1 核心理念

**简单 ≠ 封死演化路径。** 在保持单线程、主+子 Agent、工具+压缩这三大前提下，预留挂钩点。

### 8.2 关键 Hook 点

**Turn 级别**：
- `onTurnStart(context)` - Turn 开始时
- `onTurnEnd(context)` - Turn 结束时（可做日志、统计、自动 todo 更新）

**工具级别**：
- `onToolStart(toolContext)` - 工具开始执行
- `onToolEnd(toolContext, result)` - 工具执行结束（可接 UI 展示、配额统计、审计日志）

**压缩级别**：
- `onCompressStart(info)` - 开始压缩
- `onCompressEnd(info, result)` - 压缩结束（调试策略、分析 summary 质量）

**状态转换**：
- `onStateTransition(newState, context)` - 状态变化时

### 8.3 好处

- 核心 event loop 不需要改
- 未来功能只要往 hook 里"插"
- 不会污染主逻辑，保持简洁

## 九、优化检查清单

### 9.1 架构层面

- [ ] 主循环使用显式状态机，而非隐式控制流
- [ ] 每个状态有清晰的职责边界和输入输出定义
- [ ] 工具系统实现完整的 6 阶段流水线
- [ ] 所有工具调用都有统一的参数验证和权限检查
- [ ] 上下文压缩使用结构化摘要（4 段式），而非简单截断
- [ ] SubAgent 有严格的生命周期管理和结果摘要机制
- [ ] 并发 SubAgent 数量有硬限制（建议 3-5 个）
- [ ] 完整的日志和度量系统，每个关键操作都可观测

### 9.2 性能层面

- [ ] 设置 token 使用阈值：60% 警告、80% 错误、92% 自动压缩
- [ ] 工具执行有超时控制（根据工具类型设置不同阈值）
- [ ] 支持用户中断当前操作（AbortSignal 机制）
- [ ] 按"任务块"压缩，而非按消息时间顺序
- [ ] SubAgent 结果只返回结构化摘要，不合并完整对话
- [ ] 批量操作工具（MultiEdit）优先于多次单文件操作

### 9.3 稳定性层面

- [ ] 每个状态转换都有错误捕获和恢复机制
- [ ] 工具失败有降级策略（重试、替代方案、优雅失败）
- [ ] 压缩失败时回退到原始消息，不影响主流程
- [ ] SubAgent 失败不传播到主 Agent，只报告摘要
- [ ] 内存压力有检测和应对策略
- [ ] 关键操作有审计日志

### 9.4 可扩展性层面

- [ ] 使用 Hook 系统预留扩展点，不修改核心逻辑
- [ ] 工具注册是动态的，支持运行时添加新工具
- [ ] 权限检查支持自定义 Hook
- [ ] 日志系统支持多 exporter（文件、控制台、远程）
- [ ] 压缩策略可配置（阈值、提示模板、摘要长度）

## 十一、总结

### 核心设计原则

| 模块 | 关键决策 | 收益 |
|------|---------|------|
| **主循环** | 显式状态机 | 可预测、易调试、易扩展 |
| **工具系统** | 统一 6 阶段管线 | 配置级添加新工具、标准化错误处理 |
| **SubAgent** | 限制用途 + 严格生命周期 | 防止上下文爆炸、结果可控 |
| **上下文压缩** | 结构化 + 有阈值策略 | 智能保留关键信息、避免简单截断 |
| **工具选择** | 小而硬的 heuristics | 不需要复杂 Planner、效果足够好 |
| **稳定性** | 完整日志 + Hook | 可观测、可调试、可渐进优化 |

### 关键成功因素

1. **把你的单线程 Agent 当成一个小操作系统来打磨**
2. **主干路径必须干净、稳定、可观测**
3. **在简单架构里预留扩展点，但保持核心简洁**
